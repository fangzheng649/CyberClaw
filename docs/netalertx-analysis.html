NetAlertX（原 Pi.Alert）是一个开源的网络设备发现、监控和告警平台。核心定位：持续发现网络中的所有设备，监控在线/离线状态，异常时即时通知。

与 CyberClaw 的关键区别：NetAlertX 是被动+主动混合的设备发现工具，不做漏洞扫描或攻击检测，但在"设备发现"和"状态监控"两个维度做得非常深。

2. 架构分析

2.1 整体架构：双进程模型

主进程
__main__.py
无限循环：扫描调度→插件执行→事件处理→通知
⟷
SQLite
app.db
共享数据库，WAL 模式
⟷
API 进程
api_server_start.py
Flask: REST + GraphQL + SSE + MCP

两个进程通过 SQLite 和 JSON 文件通信。API 层不直连数据库，而是读主进程"物化"好的 JSON 文件（/tmp/api/table_*.json），简化了并发问题。

2.2 主循环流程

while True:
  1. importConfigs()              ← 重新加载配置和插件
  2. run_plugin_scripts("once")   ← 启动时执行一次
  3. check_and_run_user_event()   ← 处理前端用户事件
  4. update_api()                 ← 更新 API JSON 文件
  5. if 1分钟已过:
     a. run_plugin_scripts("schedule")            ← 定时扫描
     b. run_plugin_scripts("always_after_scan")   ← 扫描后处理
     c. process_scan()                            ← 扫描结果入库
     d. run_plugin_scripts("on_new_device")       ← 新设备触发
     e. run_plugin_scripts("before_name_updates") ← 名称更新前
     f. update_devices_names()                    ← DNS/mDNS 解析
     g. get_notifications()                       ← 生成通知
     h. run_plugin_scripts("on_notification")     ← 通知触发
  6. WorkflowManager.process()    ← 工作流引擎
  7. sleep(5秒)

3.2 字段来源追踪机制

每个可被插件更新的字段都有一个对应的 devXxxSource 列，值可以是：

来源值含义
USER用户手动设置，任何插件不可覆盖
LOCKED锁定状态，等同 USER
NEWDEV新设备创建时设置，首次扫描后可被覆盖
ARPSCANARP 扫描插件设置，其他插件可按优先级覆盖
NMAPDEVNmap 设备扫描设置

这个机制解决了多插件对同一字段的竞争问题。CyberClaw 的 12 个 MCP Server 缺少类似机制，可能导致数据冲突。

4. 插件扫描系统

4.1 插件架构

每个插件目录包含：

插件通过 Plugin_Objects 基类输出标准格式的管道数据（| 分隔），主进程统一解析。

4.3 核心扫描插件详解

arp_scan（主力扫描器）

Layer 2 推荐

  • 调用 arp-scan 发送 ARP 请求
  • 支持多子网、多接口、持续扫描模式
  • 输出：MAC + IP + 厂商
  • 支持 VLAN 接口

适用：发现同网段所有设备，包括沉默设备

nmap_dev_scan（深度扫描）

Layer 2

  • 调用 nmap -sn 进行主机发现
  • 解析 XML 输出获取 hostname、厂商
  • 支持为无 MAC 设备生成假 MAC

适用：获取更丰富的设备信息

icmp_scan（在线检测）

Layer 3

  • ping 模式：逐个 ping 数据库中的设备
  • fping 模式：批量扫描，支持 CIDR
  • 检测设备在线/离线

适用：快速检测设备状态变化

路由器集成（6种）

API 接入

  • Fritz!Box — TR-064 协议
  • MikroTik — RouterOS API
  • Asuswrt — 异步 API
  • Omada SDN — TP-Link 控制器
  • UniFi — Ubiquiti API
  • Pi-hole — REST API

适用：从路由器/DHCP 直接获取设备列表

4.4 设备识别与指纹

多级设备属性猜测

device_heuristics.py 的 guess_device_attributes() 按优先级匹配：

  1. 严格 MAC+Vendor 匹配 — MAC 前缀 + 厂商名组合
  2. 设备名正则匹配 — 通过名称模式（如 "iPhone"、"Galaxy"）
  3. 随机 MAC 检测 — 跳过本地管理地址
  4. 宽松 Vendor 匹配 — 仅厂商名子串
  5. IP 地址匹配 — 网关 IP = 路由器

规则存储在 back/device_heuristics_rules.json，支持覆盖。

名称解析链

DNS DIG → mDNS/Bonjour → nslookup → NetBIOS，按优先级依次尝试。

7. MCP 集成

NetAlertX 也实现了 MCP 协议（mcp_endpoint.py），但方式与 CyberClaw 不同：

维度NetAlertXCyberClaw
MCP 方式HTTP/SSE + JSON-RPC 2.0，包装现有 REST APIFastMCP stdio，每个工具独立进程
工具定义从 OpenAPI 规范自动转换手动用 @mcp_tool 装饰器
工具执行loopback HTTP 调用内部 API直接函数调用
会话管理5 分钟超时自动清理无会话概念

NetAlertX 的"MCP 包装 REST"模式更灵活——新增 REST 端点后 MCP 自动暴露，无需额外代码。

9. 关键借鉴点

9.1 CyberClaw 应该从 NetAlertX 学习什么

P0：SQLite 数据持久化

NetAlertX 做法：SQLite + WAL 模式，10 张表 + 6 个视图 + 24 个索引。主进程写 DB，API 读物化 JSON。

CyberClaw 应做：替换内存 deque 为 SQLite。最小改动——在现有 service 类中加 DB 写入。

P0：真实网络设备发现

NetAlertX 做法：18 种扫描源，插件管道统一输出格式，主循环去重比对。

CyberClaw 应做：实现 ARP 扫描 + nmap 发现。新设备自动入库替代 topology.json。

P1：多源扫描插件体系

NetAlertX 做法：每个扫描器独立目录，config.json 定义元数据，Plugin_Objects 统一输出格式。

CyberClaw 应做：统一 12 个 MCP Server 的输出格式，支持注册新扫描源。

P1：字段来源追踪

NetAlertX 做法：每个字段有 devXxxSource 列，USER/LOCKED 不可被插件覆盖。

CyberClaw 应做：多 MCP Server 对同一设备属性写入时需要冲突解决机制。

P2：工作流引擎

NetAlertX 做法：Trigger + Condition + Action 三段式，JSON 配置。

CyberClaw 应做：实现 "告警→扫描→CVE查询→隔离建议" 的自动化链。

P2：API 认证 + Bearer Token

NetAlertX 做法：自动生成 Token，常量时间比较，支持 Header/QueryParam。

CyberClaw 应做：添加 JWT 或 Bearer Token 认证。

9.2 CyberClaw 不应该照搬什么

  • PHP 前端 — NetAlertX 用 PHP + jQuery，这是 2015 年的技术选型。CyberClaw 的 Three.js + Vite 更现代。
  • 同步 Flask — NetAlertX 用同步 Flask，CyberClaw 的 FastAPI 异步架构更适合实时场景。
  • 管道文件通信 — NetAlertX 用 | 分隔的文本文件在插件和主进程间通信，CyberClaw 的 MCP stdio 协议更规范。
  • 仅在线/离线检测 — NetAlertX 不做漏洞扫描、威胁检测、安全分析。CyberClaw 在安全检测维度更强。

9.3 差异化路线建议

CyberClaw 的独特定位：AI Agent 驱动的 IoT 安全实验室

NetAlertX 擅长设备发现和状态监控，但不做安全分析。CyberClaw 应该：

  1. 借鉴 NetAlertX 的设备发现能力（ARP 扫描 + 多源发现 + 持久化存储）
  2. 保留 CyberClaw 的安全分析优势（Suricata IDS + CVE 查询 + 基线检查）
  3. 强化 CyberClaw 的差异化杀手锏（MCP 工具链 + AI Agent + 3D 可视化）

最终目标：用户说"分析摄像头 Camera-HK-1 的安全状态" → AI 自动调 nmap 扫描 + CVE 查询 + Suricata 关联 + 基线检查 → 返回综合分析报告并推荐响应动作。目前市场上没有任何开源项目能做到这一点。
