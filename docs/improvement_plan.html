CyberClaw 改进方案

针对当前五项不足，给出可落地的技术方案与实施路径。
定位：IoT 专项安全运营工具 — 填补传统 SIEM/IDS 管不到的 IoT 盲区。

1. 真实流量检测 P0 2. HUD 实用化 P0 3. MITRE 场景框架 P2 4. AI 威胁分析 P1 5. 产品定位 P2 实施优先级
1

真实流量检测 — 集成 Suricata

P0 优先
当前问题 不像 Suricata/Zeek 能抓包分析，CyberClaw 的"检测"完全依赖外部扫描和 syslog 上报，自身没有 DPI 能力。无法检测正在发生的网络攻击（端口扫描、暴力破解、异常流量）。
改进方案 CyberClaw 不需要自己造 DPI 轮子。正确做法是嵌入 Suricata 作为检测引擎，CyberClaw 负责解析告警、关联分析、触发响应。

┌──────────────┐    ┌──────────────┐    ┌────────────────┐
│ 网络镜像 /   │    │   Suricata   │    │   eve.json     │
│ SPAN 端口 /  │───→│   (IDS 模式) │───→│   实时事件流    │
│ PCAP 文件    │    │              │    │                │
└──────────────┘    └──────────────┘    └───────┬────────┘

                                    │ suricata_service.py  │
                                    │                      │
                                    │  解析 eve.json →     │
                                    │  映射 5-state FSM →  │
                                    │  WebSocket 推送 HUD  │
                                    └──────────────────────┘

实施步骤

  1. 轻量抓包层：用 scapy.sniff() 或 pyshark（tshark 封装）做基础流量统计 — 协议分布、连接计数、异常流量检测
  2. 集成 Suricata：后端用 watchdog 库监听 eve.json 文件变化，实时解析 alert/dns/http/anomaly 事件
  3. 事件映射：新建 server/services/suricata_service.py（约 200 行），将 Suricata 告警映射到 CyberClaw 的 5-state FSM（secure → scanning → vulnerable → attacked → isolated）
  4. WebSocket 推送：告警实时推送到 HUD，触发设备状态变化和告警列表更新
  5. IoT 专项规则：编写 Suricata 规则集覆盖 IoT 场景 — 摄像头 RTSP 暴力破解、MQTT 未授权访问、Modbus 异常读写、UPnP 漏洞利用

投入评估：中等 — Suricata 规则配置是核心工作，Python 集成约 200 行。Suricata 支持 Windows/Linux。

HUD 实用化 — 保留 3D + 增加 SOC 面板

P0 优先
当前问题 3D 拓扑球视觉效果好，但安全运维人员需要的是告警列表、趋势图、日志检索。真正的 SOC 用 TheHive/OpenSearch 这种界面。当前 HUD 无法支撑日常安全运维工作。
改进方案 3D 拓扑保留（作为网络概览），但必须在 HUD 左侧和下方增加三个实用 SOC 面板：告警列表面板、趋势图面板、日志检索面板。引入 ECharts 做数据可视化。

实施步骤

  1. 引入 ECharts：npm install echarts（轻量图表库，无需 React/Vue）
  2. 告警列表面板：按时间倒序展示安全事件，支持按 severity / device / type 三维过滤，点击跳转设备详情
  3. 趋势图面板：24 小时告警数量折线图、设备状态分布饼图、协议流量柱状图、攻击类型 TOP10
  4. 日志检索面板：可搜索的表格，统一检索 syslog / snmp trap / mqtt message，支持全文搜索 + 多字段过滤
  5. 布局调整：3D 拓扑缩小到左上 1/4 区域，其余空间给实用面板。支持面板折叠/展开
  6. 数据源：WebSocket 实时推送 + /api/security/events 查询历史事件

Scenario 改为 MITRE ATT&CK 演练框架

P2 优先
当前问题 攻击场景模式是写死的逐步推进动画，本质是演示而非真实的攻击检测。对安全评估没有实际价值。
改进方案 改为基于 MITRE ATT&CK for ICS 的检测验证框架。每个场景是真实的攻击操作 + 真实的检测验证，输出检测覆盖率报告。

AI 威胁分析 — 真正的安全推理

P1 优先
当前问题 chat 接口接的是后端 API，不是真正的威胁分析 AI。目前只是工具调用入口，输入"最近有什么异常"只会返回工具列表，而不是给出分析结论。
改进方案 利用已有的 GLM-4 API，构建三层 AI 分析能力：事件分类 → 事件关联 → 报告生成。将 chat 从工具入口改为安全分析助手。 
┌────────────────────────────────────────────────────────┐
│  AI 安全分析引擎                                        │
│                                                        │
│  Layer 1: 事件分类                                    │
│  ┌──────────────────────────────────────────────────┐  │
│  │ 输入: 单条 syslog / snmp trap / suricata alert   │  │
│  │ 输出: severity + attack_type + mitre_technique   │  │
│  │ 触发: 每条新事件自动执行                          │  │
│  └──────────────────────────────────────────────────┘  │
│                        │                               │
│                        ▼                               │
│  Layer 2: 事件关联                                    │
│  ┌──────────────────────────────────────────────────┐  │
│  │ 输入: 时间窗口内的多个分类后事件                   │  │
│  │ 输出: 攻击链还原 + 关联分析 + 影响评估            │  │
│  │ 触发: 每小时 / 事件数阈值 / 用户手动              │  │
│  └──────────────────────────────────────────────────┘  │
│                        │                               │
│                        ▼                               │
│  Layer 3: 报告生成                                    │
│  ┌──────────────────────────────────────────────────┐  │
│  │ 输入: 时间范围 + 关联结果                         │  │
│  │ 输出: 安全态势报告（自然语言 + 统计数据）          │  │
│  │ 触发: 用户提问 / 定时任务                         │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘

实施步骤

  1. 事件分类服务：server/services/ai_analyzer.py — 收到 syslog/snmp trap/suricata alert 后，自动调用 GLM-4 判断严重度和攻击类型
  2. 事件关联引擎：跨设备、跨协议关联多个告警，还原攻击链。例如：端口扫描(A) → 暴力破解(B) → 异常登录(C) 三条告警关联为一次攻击
  3. Chat 改造：POST /api/ai/analyze 和 POST /api/ai/correlate — 输入"最近一小时有什么异常"→ 返回 AI 分析结果而非工具列表
  4. 报告生成：POST /api/ai/report — 每日/每周自动生成安全态势报告（Markdown 格式）

投入评估：中等 — 核心是 prompt engineering + 事件聚合逻辑。差异化卖点，提升学术价值。

产品定位 — IoT 专项补充，而非替代

P2 优先
当前问题 无法替代任何成熟安全产品 — Snort/IDS、SIEM（Splunk/ELK）、漏洞扫描（Nessus）都做得更好。试图"全做"只会全做不好。
改进方案 不替代，做补充。 CyberClaw 的真正价值是填补传统 SIEM/IDS 管不到的 IoT 盲区。摄像头、传感器、智能插座这些设备通常不在企业安全体系管理范围内。

CyberClaw vs 传统安全产品

能力维度 Splunk/ELK Suricata/Snort Nessus CyberClaw
IoT 设备发现 - - 弱 强
IoT 协议理解 - 部分 - MQTT/CoAP/Modbus
默认密码检测 - - 部分 专项
设备级隔离 - - - iptables/交换机端口
通用日志分析 强 - - 弱
网络层检测 - 强 - 依赖 Suricata
漏洞扫描深度 - - 强 浅（CVE 查询 + 基线）

SIEM 联动接口

  1. 输出告警：CyberClaw 安全事件以 CEF（Common Event Format）/ JSON 格式发送到 Splunk/ELK，纳入统一 SIEM 分析
  2. 接收规则：从 SIEM 接收 IoT 相关的检测规则，动态更新 Suricata 规则集
  3. API 联动：提供 REST API 供 SIEM 调用设备隔离、配置审计等操作
投入评估：低 — 主要是架构设计和接口定义，代码量少但影响全局。
